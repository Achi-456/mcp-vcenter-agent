# Phase 4 — LLM Answer Generation

> Goal: Replace the current hardcoded answer formatter with real LLM-based answer generation while keeping a safe fallback formatter when no LLM key/provider is configured.

---

## 1. Problem Statement

The current assistant can call tools and receive tool results, but the final answer is still mostly generated by hardcoded formatting logic.

This causes these issues:

| Issue | Impact |
|---|---|
| Hardcoded response templates | Answers feel robotic and cannot adapt to complex tool outputs |
| Empty results produce misleading tables | User may think an object was found even when tool returned no valid data |
| Tool errors are not explained well | User cannot understand what failed and what to check next |
| Old local agent had better report-style responses | New Kubernetes agent needs same output quality but safer architecture |
| No LLM synthesis from multiple tool results | Multi-step queries cannot be summarized intelligently |

Phase 4 fixes this by using an LLM to generate the final assistant response from:

```text
user message + intent + tool trace + tool results JSON + safety result
```

The LLM must not call vCenter directly. It only receives already-collected tool results and produces a clean final answer.

---

## 2. What Phase 4 Must Deliver

At the end of Phase 4, this flow should work:

```text
User asks question
        ↓
Agent classifies intent
        ↓
Safety check runs
        ↓
Read-only tool executes
        ↓
Tool returns structured JSON
        ↓
LLM receives user message + tool result JSON
        ↓
LLM generates final human-friendly answer
        ↓
Frontend streams final answer
        ↓
If LLM is not configured, fallback formatter is used
```

Expected behavior:

```text
✅ LLM generates final answers from real tool result JSON
✅ Hardcoded formatter is used only as fallback
✅ Empty tool results are explained honestly
✅ Tool failures are explained clearly
✅ Suggested next steps are generated
✅ No secret values are sent to LLM
✅ No dangerous actions are generated or implied
✅ Read-only Phase 1.4 safety still applies
```

```text

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓
```

Important:

```text

LLM must not invent tool results.
```

## 6. LLM Context Payload

Create a clean context object for answer generation.

Example:

```json
{
  "user_message": "get details for esxi01.dclab.com",
  "intent": "get_host_details",
  "target_type": "host",
  "target_name": "esxi01.dclab.com",
  "safety": {
    "risk_level": "read_only",
    "blocked": false
  },
  "tool_trace": [
    {
      "tool": "get_host_details",
      "status": "success",
      "summary": "Found ESXi host esxi01.dclab.com"
    }
  ],
  "tool_results": [
    {
      "tool": "get_host_details",
      "status": "success",
      "data": {
        "name": "esxi01.dclab.com",
        "connection_state": "connected",
        "power_state": "poweredOn",
        "version": "7.0.3",
        "cpu_cores": 32,
        "memory_gb": 256,
        "vm_count": 42,
        "cluster": "Core-Infra-Cluster"
      }
    }
  ]
}
```

Before sending this to the LLM, run redaction.

Do not include:

```text
VCENTER_PASSWORD
LLM_API_KEY
Authorization headers
Kubernetes Secret values
Raw session cookies
Private keys
```

## 7. System Prompt

Create a prompt file:

```text

```

Recommended system prompt:

```text
You are a read-only vCenter operations assistant for an infrastructure engineering dashboard.

Your job is to explain vCenter inventory, ESXi hosts, VMs, datastores, networks, alarms, and events using the tool results provided to you.

Rules:
1. Use only the provided tool results as facts.
2. Do not invent VM, host, datastore, alarm, or event data.
3. If tool results are empty, clearly say that no matching object or data was found.
4. If a tool failed, explain what was attempted and summarize the error in user-friendly language.
5. Always state that no action was taken for read-only inspections.
6. Never claim that you powered on, powered off, deleted, migrated, rebooted, or modified anything.
7. High-risk actions are disabled in this phase and require approval workflow later.
8. Keep answers concise but useful.
9. Use tables for object details when helpful.
10. Include a short “Suggested next step” when useful.

Answer style:
- Professional infrastructure engineer tone.
- Mention exact object names from the tool result.
- Use Markdown tables for VM/host/datastore details.
- For errors, include: Objective, What I tried, Result, Suggested next step.
```

## 11. Fallback Formatter

Create fallback formatter:

```text
apps/engine/app/formatters/fallback_formatter.py
```

Fallback is used when:

```text
LLM key not configured
LLM provider unavailable
LLM request times out
LLM returns error
```

Fallback must be honest and safe.

Example behavior for host details:

```text
I found ESXi host esxi01.dclab.com. No action was taken.

| Property | Value |
|---|---|
| Connection State | connected |
| Power State | poweredOn |
| Version | 7.0.3 |
| CPU Cores | 32 |
| Memory | 256 GB |
| VM Count | 42 |
| Cluster | Core-Infra-Cluster |

Suggested next step:
I can show VMs running on this host, check recent events, or summarize active alarms related to it.
```

Example behavior for empty result:

```text
I could not find a matching object for `unknown-object-name`. No action was taken.

What I tried:
- Searched the vCenter inventory using the selected tool.
- No matching VM, host, datastore, network, or cluster was returned.

Suggested next step:
Check the exact object name from the Inventory page, or ask me to list available hosts or VMs.
```

Fallback must never return fake values like:

```text
Power State: unknown
CPU: 0
Memory: 0 GB
```

unless those values truly came from the tool result.

## 12. Empty Tool Result Handling

Add explicit logic:

```python
if tool_result.status == "success" and data_count == 0:
    result_type = "empty"
```

LLM should be told:

```text
The tool succeeded but returned zero matching items.
Explain that no matching object/data was found.
Do not create a details table with default values.
```

For object lookup:

```text
VM_NOT_FOUND → say VM was not found
HOST_NOT_FOUND → say host was not found
WRONG_OBJECT_TYPE → say object appears to be another type and suggest correct tool
```

## 13. Tool Error Handling

If tool failed, LLM context should include:

```json
{
  "tool": "get_host_details",
  "status": "error",
  "error_code": "VCENTER_SESSION_EXPIRED",
  "message": "vCenter session expired after retry."
}
```

LLM answer format:

```text
I could not retrieve the requested details. No action was taken.

Objective:
Get details for ESXi host esxi01.dclab.com.

What I tried:
- Called get_host_details.
- Reconnected to vCenter and retried once.

Result:
The vCenter session expired or authentication failed.

Suggested next step:
Check the vCenter credential status in Settings, then retry the request.
```
